## 1. 제품 개요

### 1.1 해결하고자 하는 문제

요구공학 초기 단계(요구 수집 단계)에서 발생하는 아래의 고질적인 문제들을 해소한다.

| 문제 | 현재 상황 | 기대 효과 |
|------|-----------|-----------|
| **이해관계자 파악 부재** | 프로젝트에 영향을 미치는 이해관계자를 비체계적으로 관리; 누락·중복 발생 | 이해관계자 전원 등록·분류, 영향력 시각화로 우선순위 명확화 |
| **인터뷰 준비 부실** | 인터뷰 목적·범위 불명확, 질문지 없이 진행하거나 개인 역량에 의존 | LLM 기반 맞춤형 질문지 자동 생성으로 준비 품질 일관화 |
| **결과 정리 비효율** | 인터뷰 후 결과가 메모·이메일에 흩어져 구조화되지 않음 | 결과 구조화 정리, 요구사항 자동 매핑으로 재활용성 확보 |
| **요구사항 연결 누락** | 인터뷰에서 수집한 내용이 요구사항 목록과 연결되지 않음 | 인터뷰 내용과 요구사항 ID 간 추적 관계 자동 생성 |
| **액션아이템 누락** | 인터뷰 중 도출된 후속 조치 사항이 관리되지 않고 소실 | 액션아이템 자동 탐지 및 등록으로 후속 관리 강화 |

### 1.2 제품(모듈) 목적

**RETS-MODULE-02**는 요구공학 초기 단계의 요구 수집 활동을 전 주기 지원하는 AI 보조 웹 도구이다. 이해관계자 식별부터 인터뷰 계획, 질문지 생성, 결과 기록, 요구사항 매핑까지 하나의 인터페이스에서 처리함으로써 인터뷰 품질의 편차를 줄이고, 수집 근거와 의사소통 내역의 정리·재활용성을 강화한다.

## 3. 유저 스토리 (RFP 기반 상세화)

### US-01: 인터뷰 대상 요구사항 목록 로드

> **As a** 요구분석가  
> **I want to** RFP 요구사항 목록(JSON)을 시스템에 불러와 인터뷰에서 다룰 요구사항을 선택할 수 있도록  
> **So that** 인터뷰 계획 수립 시 요구사항과의 연결 관계를 처음부터 구조적으로 설정할 수 있다

**수락 기준 (Acceptance Criteria):**
- JSON 파일을 불러오기 버튼으로 로드하면 요구사항 목록이 그리드에 표시된다
- 요구사항 목록에는 인터뷰 내용을 매핑하기 위한 전용 컬럼이 존재한다
- 인터뷰 계획 생성 시 요구사항 목록에서 N개를 선택하여 계획과 연결할 수 있다
- 로드된 스키마는 `rets_02c_req_schema.json` 형식을 따른다

---

### US-02: 이해관계자 식별·등록 및 영향도 분류

> **As a** 프로젝트 관리자  
> **I want to** 프로젝트 관련 이해관계자를 속성과 함께 등록하고, 영향력 기준으로 분류·시각화할 수 있도록  
> **So that** 핵심 이해관계자를 우선 식별하고 인터뷰 우선순위를 결정할 수 있다

**수락 기준:**
- 이해관계자를 이름, 역할, 조직, 연락처 속성과 함께 등록·수정·삭제할 수 있다
- 등록된 이해관계자를 Power/Interest 기준의 매트릭스 차트로 시각화한다
- 이해관계자 목록은 필터링 및 정렬이 가능하다

---

### US-03: 인터뷰 계획 수립 및 일정 관리

> **As a** 요구분석가  
> **I want to** 이해관계자별 인터뷰 계획을 등록하고 전체 인터뷰 일정을 한눈에 확인할 수 있도록  
> **So that** 인터뷰 일정 충돌을 방지하고 진행 현황을 관리할 수 있다

**수락 기준:**
- 인터뷰별로 목적, 관련 요구사항 범위(목록), 참여자, 일시, 장소, 예상 소요시간을 등록할 수 있다
- 전체 인터뷰 일정을 캘린더 뷰 또는 목록 뷰로 전환하여 확인할 수 있다
- 인터뷰 이력을 통합 조회하고, 과거 인터뷰 내용을 재활용할 수 있다

---

### US-04: AI 기반 인터뷰 질문지 초안 생성 (LLM)

> **As a** 요구분석가  
> **I want to** 등록된 인터뷰 계획을 바탕으로 LLM이 맞춤형 질문지 초안을 자동 생성해 주도록  
> **So that** 질문 설계에 소요되는 시간을 줄이고, 이해관계자별로 최적화된 질문을 준비할 수 있다

**수락 기준:**
- 인터뷰 계획(이해관계자 역할, 업무영역, 요구사항 범위)을 LLM에 전달하면 질문지 초안이 자동 생성된다
- 생성된 질문지를 항목 단위로 추가·수정·삭제할 수 있다
- 인터뷰별로 질문지 버전을 저장·관리할 수 있다
- LLM이 제안한 질문 각각에 대해 수락/거부 처리가 가능하다

---

### US-05: 인터뷰 결과 구조화 정리, 요구사항 매핑, 액션아이템 도출 (LLM)

> **As a** 요구분석가  
> **I want to** 인터뷰에서 기록한 내용을 LLM이 자동으로 구조화하고, 요구사항과 매핑하며, 액션아이템을 추출해 주도록  
> **So that** 인터뷰 후 정리 작업 부담을 줄이고 요구사항 추적성을 즉시 확보할 수 있다

**수락 기준:**
- 인터뷰 내용 입력 후 LLM이 항목별 채번과 함께 구조화 정리 결과를 제안한다
- 요구사항 ID가 명시적으로 언급되지 않아도 LLM이 추론하여 관련 요구사항 매핑을 제안한다
- 매핑된 요구사항에 대해 사용자가 수락/거부 처리할 수 있다
- 액션아이템으로 판단되는 내용에 대해 시스템이 액션아이템 등록을 제안한다
- 최종 확정된 인터뷰 결과를 Markdown 파일로 내보낼 수 있다
- 요구사항 매핑 결과를 CSV 파일로 내보낼 수 있다

## 4. 제품(모듈) 범위

## 6. 사용자 경험 및 디자인 고려사항

### 6.1 UX 원칙

- **단계적 흐름 지원:** 요구사항 로드 → 이해관계자 등록 → 인터뷰 계획 → 질문지 준비 → 결과 입력 → 출력 순서의 자연스러운 워크플로우 제공
- **AI 투명성:** LLM이 제안한 내용은 시각적으로 구분(배지·아이콘)하며, 사용자가 반드시 수락/거부 결정을 내리도록 설계
- **대시보드 우선:** 전체 현황을 홈 화면에서 즉시 파악 가능하도록 대시보드 화면 제공
- **오류 친화성:** LLM 응답 실패 시 명확한 오류 메시지와 수동 입력 fallback 제공

### 6.2 디자인 시스템

- RETS 공통 디자인 시스템 준수 (`rets_02b_design_requirements.md`, `rets_02b_design_system_ref.html`)
- 한국어/영어 다국어 UI 전환 지원
- 반응형 레이아웃 (데스크톱 우선, 태블릿 대응)

### 6.3 주요 화면 구성

| 화면 | 설명 |
|------|------|
| 대시보드 | 이해관계자 수, 인터뷰 현황, 미완료 액션아이템 요약 |
| 이해관계자 관리 | 목록 그리드 + Power/Interest 매트릭스 차트 |
| 인터뷰 계획 | 캘린더 뷰 / 목록 뷰 전환, 계획 등록 폼 |
| 질문지 편집 | 질문 카드 리스트, LLM 생성 버튼, 항목 편집 |
| 인터뷰 결과 | 결과 입력 에디터, LLM 구조화 결과 패널, 요구사항 매핑 테이블 |
| 파일 출력 | Markdown / CSV 다운로드 버튼 |

## 7. 기술 고려사항

### 7.1 구현 제약

| 항목 | 내용 |
|------|------|
| 구현 방식 | Vanilla JS, 단일 HTML 파일 (`rets_module02_07_app.html`) |
| 백엔드 | 없음 (프론트엔드 전용 PoC) |
| 저장소 | 브라우저 메모리 또는 localStorage (세션 내 데이터 유지) |
| 외부 라이브러리 | 필요 시 CDN 활용 (Chart.js 등, 단일 HTML 내 통합) |
| 파일명 규칙 | `rets_module02_07_app.html` |

### 7.2 LLM 연동

| 항목 | 내용 |
|------|------|
| LLM 모델 | Google Gemini (LLM 릴레이 서버 경유) |
| API 문서 | `https://ai-relay.mooo.com:11443/openapi.json` (또는 로컬 `rets_02a_llm_api_openapi_doc.json`) |
| 개발 가이드 | `/00_rfp_requirements/rets_02a_llm_relay_server_dev_guide.md` |
| 호출 흐름 | 세션 초기화(시스템 프롬프트) → 채팅 계속 |
| 요청 형식 | `team_id` + `message` (구조화 JSON 반환 요청 포함) |
| 응답 처리 | 응답 텍스트 내 JSON 파싱; 파싱 실패 시 오류 처리 |
| 비동기 처리 | LLM 호출은 전부 async/await 비동기 처리, 로딩 인디케이터 표시 |
| 사용자 개입 | 모든 LLM 제안에 수락/거부 UI 필수 제공 |

### 7.3 데이터 스키마

- 입력 데이터: `rets_02c_req_schema.json` 준수 (RFP 요구사항 목록 JSON)
- 출력 데이터:
  - 인터뷰 결과: Markdown 형식 (형식 자율)
  - 요구사항 매핑: CSV 형식 (req_id, interview_id, mapping_content, confidence 등 포함)

### 7.4 배포

- GitHub Pages 배포 (Public URL 획득)
- 배포용 파일명: `index.html` (앱 파일 복사본)

### 8.1 개발 우선순위 (MoSCoW)

| 구분 | 기능 |
|------|------|
| **Must Have** | 요구사항 JSON 로드, 이해관계자 CRUD, 인터뷰 계획 등록, 인터뷰 결과 입력, LLM 질문지 생성, LLM 결과 구조화, 파일 출력 |
| **Should Have** | Power/Interest 매트릭스 시각화, 캘린더 뷰, 요구사항 매핑(추론 기반), 액션아이템 도출 |
| **Could Have** | 질문지 버전 관리, 인터뷰 이력 재활용, 다국어 전환 |
| **Won't Have (이번 PoC)** | 백엔드 연동, 실시간 협업, 외부 캘린더 연동 |

### 8.2 TDD 접근 방식

RFP 권장에 따라 테스트 주도 개발 방식을 채택한다.

1. MD04(테스트 설계서)에서 핵심 유저 시나리오별 테스트케이스를 먼저 정의한다
2. 정의된 테스트케이스를 충족하는 방향으로 HTML 앱을 구현한다
3. 구현 완료 후 MD08에서 Playwright 기반 자동화 테스트로 핵심 시나리오를 검증한다
